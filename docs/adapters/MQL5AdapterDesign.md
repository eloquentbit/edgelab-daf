## Purpose

This document defines the architectural design of the MQL5 Platform Adapter.

The adapter is responsible for translating MetaTrader 5 trading concepts into the canonical interaction model exposed by the Data Acquisition Framework (DAF).

The MQL5 Adapter is the reference implementation of the Platform Adapter architecture described by the Data Acquisition Framework.

Its purpose is not to expose MetaTrader internals to the framework, but to isolate every platform-specific concept behind a well-defined integration layer.

---

## Why an Adapter?

MetaTrader 5 defines its own trading model based on Orders, Deals and Positions.

The Data Acquisition Framework intentionally does not depend on these concepts.

Instead, the adapter is responsible for translating platform-specific concepts into the canonical domain model defined by the DAF.

The framework operates on a simpler and platform-independent interaction model composed of:

- Trade Opened
- Position Updated
- Trade Closed
- Feature Added

The responsibility of the adapter is therefore to translate the MetaTrader execution model into this canonical language.

---

## Responsibilities

The MQL5 Adapter is responsible for:

- Observing trading activity generated by MetaTrader 5.
- Detecting the lifecycle of trading positions.
- Translating platform events into DAF interaction events.
- Collecting all platform-specific information required to progressively build the canonical TradeRecord.
- Remaining the only component aware of MetaTrader-specific APIs and data structures.

The adapter is **not** responsible for:

- Building TradeRecords.
- Persisting data.
- Exporting datasets.
- Computing analytics.
- Managing custom strategy features.

Those responsibilities belong exclusively to the DAF Core.

---

## Concept Translation

| MetaTrader 5      | Canonical DAF Concept     |
|-----------------  |---------------------------|
| Order             | Internal platform concept |
| Deal              | Execution event |
| Position          | Trade lifecycle |
| Position Created  | Trade Opened |
| Position Modified | Position Updated |
| Position Closed   | Trade Closed |

The adapter hides the complexity of MetaTrader while exposing only the concepts required by the framework.

---

## Design Principles

### Platform Independence

The DAF Core must never depend on MetaTrader 5 types, APIs or callbacks.

### Single Responsibility

The adapter translates platform events.
It does not contain business logic related to data acquisition or persistence.

### Canonical Representation

Regardless of the execution platform, the resulting TradeRecord must have the same structure.

### Extensibility

The design should serve as the reference implementation for future adapters (MQL4, cTrader, TradeLocker and others).

Future Platform Adapters should preserve the same canonical interaction model while translating the concepts of their respective trading platforms.

### Event-Driven Interaction

The adapter reacts to trading events generated by MetaTrader 5 and translates them into the canonical interaction model understood by the DAF Core.

The adapter never reconstructs trade state through polling.

---

## Adapter Boundaries

The MQL5 Adapter is responsible only for translating platform events into the canonical interaction model.

It must never:

- perform analytics;
- compute derived metrics;
- persist data;
- expose MetaTrader-specific concepts outside the adapter boundary.

Its only responsibility is translation.

---

## Implementation Questions

The following topics will be investigated before implementation:

- Which MQL5 callback should be considered the primary source of truth (`OnTradeTransaction()` or `OnTrade()`)?
- How should partial closes be represented?
- Which platform events should generate `Position Updated`?
- Which platform fields are mandatory to initialize a TradeRecord?
- How should netting and hedging accounts be handled?

These questions will be answered during the MQL5 domain analysis before implementation begins.
