# ADR-003 — Core Fields and Feature Set

| Metadata | Value |
|----------|-------|
| **Status** | Accepted |
| **Date** | 2026-06-28 |

---

# Context

The Data Acquisition Framework is intended to support trading strategies with widely different objectives and methodologies.

While every executed trade shares a common set of platform-independent properties, each strategy may also produce additional contextual information that is relevant only to its own trading logic.

Examples include:

- ATR
- Opening Range Size
- Relative Volume
- Gap Percentage
- Market Regime
- Session
- Volatility Metrics

The framework therefore requires a representation capable of separating universally applicable trading information from strategy-specific knowledge.

---

# Decision

A `TradeRecord` is divided into two independent sections:

- **Core Fields**
- **Feature Set**

Core Fields contain the canonical information that every trade should expose regardless of the execution platform or trading strategy.

The Feature Set contains optional strategy-specific information recorded during the trade lifecycle.

Both sections belong to the same `TradeRecord`, but they serve different responsibilities.

---

# Rationale

Separating Core Fields from the Feature Set provides a clear distinction between:

- information generated by the trading platform;
- information generated by the trading strategy.

Core Fields define the canonical representation of a trade.

The Feature Set allows unlimited extensibility without modifying the framework's domain model.

This separation enables the framework to evolve while maintaining a stable and platform-independent `TradeRecord`.

---

# Consequences

## Positive

- The canonical TradeRecord remains stable.
- Strategies can record arbitrary contextual information.
- New research ideas do not require changes to the Core model.
- Different strategies remain fully compatible with the same framework.

## Negative

- Feature names should follow consistent naming conventions.
- Consumers of exported datasets must distinguish between canonical fields and optional features.

---

# Alternatives Considered

## Single Flat Structure

Storing every piece of information inside a single flat structure was considered.

This option was rejected because the canonical representation would continuously evolve as new strategies introduced additional fields.

The resulting model would become difficult to maintain and impossible to standardize.

---

## Strategy-Specific TradeRecord

Creating different `TradeRecord` structures for different strategies was also considered.

This option was rejected because it would prevent the framework from defining a canonical representation of executed trades.

One of the primary goals of the DAF is to provide a common domain model independent of individual strategies.
